# Virtual Debug Environment

This document describes a reproducible local debug environment for the RLS Screening App MVP.

## Goal

The virtual debug environment lets collaborators run the full MVP loop without real Apple HealthKit, Android Health Connect, or production infrastructure.

It uses:

- FastAPI backend on `http://localhost:8000`
- SQLite by default for local persistence
- React web dashboard on `http://localhost:5173`
- Mock wearable data generated by the web/mobile scaffold
- Deterministic fallback inference unless optional model dependencies are installed

## Environment Layout

```text
rls-screening-app/
  services/api/      FastAPI backend and local SQLite database
  apps/web/          React/Vite dashboard
  apps/mobile/       Flutter scaffold with mocked health sync
  docs/              architecture, API, debugging, and collaborator docs
```

## Backend Virtual Environment

Create a Python virtual environment:

```bash
cd services/api
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Run the API:

```bash
uvicorn app.main:app --reload
```

Check that it works:

```bash
curl http://localhost:8000/health
```

Expected behavior:

- `GET /health` returns a healthy response.
- API docs are available at `http://localhost:8000/docs`.
- SQLite database is created locally as `services/api/rls_screening.db`.

The SQLite file is ignored by git.

## Web Debug Environment

Install and run:

```bash
cd apps/web
npm install
npm run dev
```

Open:

```text
http://localhost:5173
```

Debug flow:

1. Register a demo user.
2. Click `Sync mock health data`.
3. Click `Submit questionnaire + predict`.
4. Review dashboard score.
5. Review debug tabs:
   - `Prediction history`
   - `Questionnaire history`
   - `Upload/debug`

## Mobile Debug Environment

The mobile app needs Flutter SDK.

```bash
cd apps/mobile
flutter pub get
flutter run
```

If Flutter is not installed, collaborators can still test the backend and web dashboard.

The mobile scaffold currently uses `MockHealthDataService`. Real Apple HealthKit / Health Connect should be added behind the same interface.

## Docker Debug Environment

For a containerized stack:

```bash
docker compose up --build
```

This starts PostgreSQL, API, and web services using the Docker configuration.

## Optional Real Model Environment

The app works without optional model dependencies by using fallback inference.

To test the real XGBoost/TabM adapter:

```bash
cd services/api
source .venv/bin/activate
pip install -r requirements-model.txt
```

Then restart FastAPI and inspect `latest_prediction.explanation_json` in `GET /reports/latest`.

## Reset Local Debug Data

Stop FastAPI, then remove the local SQLite database:

```bash
cd services/api
rm rls_screening.db
```

Start FastAPI again. The app will recreate tables on startup.

Do not commit local database files.

## Expected Backend Log Pattern

A successful virtual debug run usually logs:

```text
POST /auth/register 200 OK
GET /reports/latest 200 OK
POST /wearable/upload 200 OK
GET /wearable/daily-features 200 OK
POST /questionnaire/submit 200 OK
POST /predict/tier2 200 OK
GET /reports/latest 200 OK
```

Browser `OPTIONS` requests are normal CORS preflight checks.
