Product idea

A focused MVP for RLS screening workflows

The current product is designed to test the full data and prediction loop before building a clinical-grade app.

01

Collect inputs

Register a demo profile, sync mock wearable events, and submit symptom questionnaire payloads.

02

Normalize features

Convert sleep, heart rate, steps, activity, and questionnaire data into backend daily features.

03

Estimate risk

Run Tier 1 or Tier 2 prediction through a real-model adapter or deterministic fallback model.

Architecture

One monorepo, clear boundaries

The scaffold separates frontend experience, mobile health-data integration, backend APIs, database models, and model inference. This keeps HealthKit work independent from backend prediction logic.

  • FastAPI + SQLModel backend
  • React web dashboard
  • Flutter mobile scaffold
  • SQLite locally and PostgreSQL in Docker
  • XGBoost/TabM adapter with fallback inference

Health data roadmap

HealthKit integration is intentionally isolated

The backend upload API is ready. The mobile app currently uses a mock HealthDataService, so a collaborator can add Apple HealthKit reads without touching model or report logic.

Collaborator scope

  • Request Apple HealthKit permissions
  • Read sleep, heart rate, resting heart rate, steps, and activity minutes
  • Normalize records into wearable events
  • Upload to POST /wearable/upload
  • Keep mock sync available for local development
